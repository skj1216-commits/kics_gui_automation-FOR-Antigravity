# KICS 권한 요청 처리 자동화 시스템

이 프로젝트는 KICS 시스템의 권한 요청 공문을 주기적으로 감시하고, 브라우저 자동화를 통해 요청된 사용자의 권한을 자동으로 부여하는 애플리케이션입니다. Python의 `tkinter`를 활용한 GUI와 `playwright`를 활용한 웹 브라우저 자동화 기술로 구현되었습니다.

## 시스템 핵심 로직 (System Logic)

시스템은 크게 **초기화**, **작업 모드(수동/감시)**, **6단계의 처리 파이프라인**, 그리고 **예외 및 이력 관리**로 구성되어 있습니다.

### 1. 실행 및 초기화
1. **GUI 및 스레드 시작**: `main.py`가 실행되면 `tkinter` 기반의 UI가 로드되고, 백그라운드 데몬 스레드로 `browser_thread_worker`(브라우저 자동화 스레드)가 시작됩니다.
2. **설정 및 DB 로드**: `config.json`에서 URL 및 스캔 설정 등을 불러오고, `database.py`를 통해 로컬 SQLite DB (`processed_docs.db`)를 초기화합니다.
3. **브라우저 실행**: `playwright`를 통해 Chromium 브라우저(headless=False)를 실행하고, 최초 로그인 등의 수동 작업을 위한 준비 상태가 됩니다.

### 2. 작업 모드
* **수동 실행 (Run Once)**: 단일 "공문 URL"을 직접 입력받아, 감시 루프를 거치지 않고 즉각적으로 문서 처리를 1회 수행합니다.
* **감시 시작 (Start Monitoring)**: 설정된 `refresh_interval_seconds`(스캔 주기)마다 "공문 목록 URL" 페이지를 새로고침하며 신규 공문을 탐색합니다. 신규 공문이 발견되면 즉시 처리 파이프라인을 가동합니다.

### 3. 처리 파이프라인 (6단계)
자동화 처리는 다음 6가지의 순차적 단계(STEP)를 거칩니다.

* **[Step 1] 공문 접속 (STEP_CONNECT)**
  - 공문 목록에서 타겟 키워드(`target_keywords` 또는 `target_title`)가 포함된 문서를 찾습니다.
  - 이전에 처리 완료(SUCCESS)된 문서인지 SQLite DB를 조회하여 중복 처리를 방지합니다.
  - 신규 문서로 판별되면, 해당 공문의 상세 페이지 URL로 이동합니다.

* **[Step 2] 데이터 추출 (STEP_EXTRACT)**
  - 공문 상세 페이지의 본문 텍스트를 읽어옵니다(Crawling).
  - **사용자 ID 추출**: 정규표현식을 사용하여 `아이디: userid` 또는 영문/숫자 조합의 ID 패턴을 추출합니다.
  - **권한 추출**: `config.json`의 `permission_catalog`에 정의된 키워드와 매칭하거나, 본문의 글머리 기호(`-`, `•` 등)를 파싱하여 요청 권한 목록을 추출합니다.
  - 추출된 정보는 즉시 UI 화면에 갱신됩니다.

* **[Step 3] 관리자 이동 (STEP_ADMIN)**
  - 권한을 부여하기 위해 설정된 "관리자 URL(`admin_url`)"로 브라우저 탭을 이동합니다.

* **[Step 4] 사용자 검색 (STEP_SEARCH)**
  - 관리자 화면 내 사용자 검색 입력창(`admin_user_search_selector`)에 Step 2에서 추출한 사용자 ID를 자동으로 입력합니다.
  - 검색 버튼(`admin_user_search_button_selector`)을 클릭하여 권한 부여 대상을 조회합니다.

* **[Step 5] 권한 선택 (STEP_SELECT)**
  - Step 2에서 추출한 권한 목록을 기반으로, 매핑된 체크박스 또는 UI 요소(`admin_permission_selector_map`)를 찾아 자동으로 클릭/선택합니다.

* **[Step 6] 최종 승인/종료 (STEP_APPROVE)**
  - 안전 장치(Safety Mechanism):
    - **Dry-run 옵션**: 켜져 있을 경우 실제 승인 버튼을 누르지 않고 완료 처리합니다. (테스트 용도)
    - **최종 승인 전 확인창**: 옵션 활성화 시, 사용자 화면에 팝업창을 띄워 관리자가 추출된 ID와 권한을 눈으로 확인하고 `Yes/No`를 선택할 수 있게 합니다.
  - 승인이 떨어지면 최종 승인 버튼(`admin_approve_button_selector`)을 클릭합니다.
  - 작업이 모두 성공적으로 끝나면 DB에 해당 공문의 처리 상태를 `SUCCESS`로 기록합니다.

### 4. 예외 처리 및 복구 로직
* **자동 재시도 (Auto Retry)**: 추출 단계 실패나 네트워크/페이지 지연(Timeout)으로 인해 오류가 발생한 경우, 설정된 `max_retries`(최대 시도 횟수)만큼 점진적 대기 시간(5초 -> 10초 -> 20초)을 두고 해당 공문의 처리를 재시도합니다.
* **실패 기록**: 재시도에도 불구하고 처리에 실패하거나, 관리자가 확인 창에서 취소(No)를 누를 경우, 작업을 중단하고 DB에 상태를 `FAILED`로 기록합니다.
* 오류 및 진행 사항은 UI 내 '처리 로그' 영역에 실시간으로 출력되며, 동시에 `logs/` 폴더 하위에 날짜별 파일로 안전하게 기록됩니다.
